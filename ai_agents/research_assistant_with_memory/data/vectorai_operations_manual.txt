VectorAI DB Operations Manual
Internal Reference Document

Chapter 1: Deployment

VectorAI DB is distributed as a container image and is intended to run alongside the application
that queries it rather than as a remote managed service, which keeps the entire retrieval path on
the local machine or private network. A standard deployment exposes three ports: a gRPC endpoint
used by client SDKs for low-latency reads and writes, a REST endpoint for tooling and health
checks, and a local web UI for browsing collections during development. Data is written to a
mounted volume so that collections survive container restarts. Operators should size the mounted
volume for at least three times the raw size of the embedding vectors being stored, to leave room
for index structures and write-ahead logs.

Chapter 2: Collections

A collection is the unit of isolation inside VectorAI DB. Each collection has a fixed vector
dimension and distance metric, set at creation time and immutable afterward; changing either
requires creating a new collection and migrating data. Cosine distance is the recommended metric
for text embeddings produced by transformer encoders, since these models are typically trained
with normalized representations where direction carries the semantic signal rather than magnitude.
Applications that need to keep unrelated types of content separate, for example raw document
chunks versus a memory store of prior findings, should use separate collections rather than a
single collection with a type field, because collection-level isolation makes it possible to clear
or rebuild one without touching the other.

Chapter 3: Writing Data

Points are written through an upsert operation, where each point carries an identifier, a vector,
and an arbitrary JSON payload. The payload is the mechanism for attaching metadata such as source
document, section heading, or chunk position, and this metadata can be read back alongside search
results without a second lookup. Operators should batch upserts rather than writing points one at a
time; batching amortizes the per-request overhead and is the dominant factor in ingestion
throughput for large document sets. There is no strict batch size limit, but batches in the range
of 32 to 128 points balance throughput against memory pressure on typical hardware.

Chapter 4: Querying

Search accepts a query vector and a result limit, and returns the nearest points under the
collection's configured distance metric along with a similarity score for each. Because the score
scale depends on the distance metric, applications that filter results by a minimum score threshold
should tune that threshold empirically against their own embedding model rather than reusing a
threshold from a different model or metric. Applications that need to inspect data without a
similarity query, for example to list every point belonging to one source document, should use the
scroll operation, which returns points directly from storage without computing similarity.

Chapter 5: Operational Notes

VectorAI DB requires no external network access once the container image has been pulled, which
makes it suitable for air-gapped or edge deployments. Collections can be deleted and recreated
cheaply, which is the recommended way to reset a memory-style collection without affecting other
collections in the same deployment. Point identifiers should be generated by the client rather than
left to autoincrement, since client-generated identifiers make it possible to upsert idempotently:
writing the same identifier twice overwrites rather than duplicates the point.
